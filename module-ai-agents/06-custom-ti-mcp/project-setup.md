# Подготовка проекта

Цель: собрать минимальный проект, в котором MCP-сервер можно запустить и подключить к OpenCode.

## Создайте каталог проекта

Эти команды создают отдельную папку и переходят в нее:

```bash
mkdir ti-mcp-server
cd ti-mcp-server
```

Ожидаемый результат: вы находитесь внутри каталога `ti-mcp-server`.

## Создайте виртуальное окружение

Виртуальное окружение изолирует зависимости этого проекта от системного Python:

```bash
python3 -m venv .venv
source .venv/bin/activate
```

Ожидаемый результат: в приглашении терминала появляется `.venv`, а Python и pip берутся из проекта.

На Windows с PowerShell активация будет выглядеть иначе:

```powershell
.venv\Scripts\Activate.ps1
```

Ожидаемый результат: PowerShell активирует окружение `.venv`.

## Установите зависимости

Эта команда ставит библиотеки, нужные серверу:

```bash
pip install fastmcp requests beautifulsoup4 python-dotenv
```

Ожидаемый результат: pip устанавливает пакеты без ошибок.

Если хотите сразу зафиксировать окружение, можно сохранить зависимости в `requirements.txt`:

```bash
pip freeze > requirements.txt
```

Ожидаемый результат: в проекте появляется или обновляется файл `requirements.txt`.

## Структура проекта

```text
ti-mcp-server/
├── server.py
├── requirements.txt
├── .env.example
├── README.md
```

## Файл `.env.example`

```text
RST_THREAT_API_KEY=put-your-api-key-here
```

Это шаблон для ключа без самого секрета.

Создайте локальный `.env` рядом с ним:

```text
RST_THREAT_API_KEY=your-real-api-key-here
```

`.env` не коммитят. Его читают локально через `python-dotenv` или через переменную окружения.

## Файл `requirements.txt`

```text
fastmcp
requests
beautifulsoup4
python-dotenv
```

## Файл `.gitignore`

```text
.venv/
.env
__pycache__/
*.pyc
ioc.json
```

## Зачем это нужно

Этот минимальный набор файлов держит проект в порядке:

- `server.py` содержит сам MCP-сервер;
- `requirements.txt` фиксирует зависимости;
- `.env.example` показывает, какие переменные нужны;
- `.gitignore` защищает от случайного коммита секретов и мусорных файлов.

## Как запустить сервер

Полный код сервера находится в [server-implementation.md](server-implementation.md).

Эти команды активируют окружение и запускают сервер:

```bash
source .venv/bin/activate
python server.py
```

Ожидаемый результат: сервер запускается без traceback и слушает локальный HTTP endpoint.

Если сервер запускается без ошибок, значит базовая часть проекта собрана правильно. В HTTP-режиме он будет доступен по адресу:

```text
http://127.0.0.1:8000/mcp
```

## Как добавить сервер в OpenCode

Добавьте `ti-tools` в `opencode.json`. Сервер подключается как `remote`, то есть как сетевой endpoint, потому что он слушает `localhost` по HTTP:

```json
{
  "$schema": "https://opencode.ai/config.json",
  "mcp": {
    "ti-tools": {
      "type": "remote",
      "url": "http://127.0.0.1:8000/mcp",
      "enabled": true
    }
  }
}
```

Включите `playwright` MCP. Он нужен, чтобы открыть статью и передать текст в `ti-tools`.

В конфиге должны быть доступны оба сервера:

- `playwright`;
- `ti-tools`.

Если нужен локальный ключ, используйте переменную окружения:

```bash
export RST_THREAT_API_KEY="your-api-key-here"
opencode
```

Ожидаемый результат: OpenCode запускается в той же сессии, где доступен API-ключ.

Так ключ не хранится в JSON.

## Как проверить, что MCP подключен

После запуска OpenCode проверьте список MCP-инструментов:

```bash
opencode mcp list
```

Ожидаемый результат: в списке есть сервер `ti-tools`.

Если список доступен, в нем должен быть `ti-tools`.

Если `ti-tools` не виден:

1. проверьте, что сервер отвечает на `http://127.0.0.1:8000/mcp`;
2. проверьте, что `RST_THREAT_API_KEY` задан;
3. перезапустите OpenCode.

## Что должно получиться

- `.env.example` объясняет, какой ключ нужен;
- локальный `.env` хранится только у вас;
- `server.py` запускается отдельно;
- `playwright` включен вместе с `ti-tools`;
- `ti-tools` добавлен в OpenCode как `remote` MCP;
- `opencode mcp list` показывает подключенный сервер.

## Практические команды

Проверить, что виртуальное окружение активировано:

```bash
which python
```

Ожидаемый результат: путь указывает на каталог `.venv`.

Проверить, что pip смотрит в нужное окружение:

```bash
python -m pip --version
```

Ожидаемый результат: путь в выводе относится к `.venv`.

Проверить список установленных пакетов:

```bash
pip list
```

Ожидаемый результат: в списке есть `fastmcp`, `requests`, `beautifulsoup4` и `python-dotenv`.

Проверить, что ключ задан:

```bash
echo "$RST_THREAT_API_KEY"
```

Ожидаемый результат: терминал выводит значение ключа или пустую строку, если ключ не задан.

Проверить, что сервер поднимается:

```bash
python server.py
```

Ожидаемый результат: сервер запускается без traceback и слушает `127.0.0.1:8000`.

Проверить HTTP endpoint:

```bash
curl -i http://127.0.0.1:8000/mcp
```

Ожидаемый результат: вы получаете HTTP-ответ от локального MCP-сервера.

## Проверка перед переходом к коду

- каталог проекта создан;
- виртуальное окружение активно;
- зависимости установлены;
- `.env.example` есть в проекте;
- в `.gitignore` нет риска случайно закоммитить ключи;
- сервер запускается командой `python server.py`;
- OpenCode видит `ti-tools`;
- локальный endpoint отвечает по `http://127.0.0.1:8000/mcp`.
