# MCP: обзор протокола и модели подключения

MCP стандартизирует подключение инструментов к агенту: агент узнает, какие действия доступны, как их вызвать и какой результат ожидать.

## Что такое MCP

MCP расшифровывается как Model Context Protocol. Это протокол для соединения AI-приложений с внешними источниками контекста, инструментами и сервисами.

Ссылки:

- [MCP docs](https://modelcontextprotocol.io/docs/learn)

## Зачем нужен MCP

Без MCP интеграции чаще пишут вручную. Это создает проблемы:

- каждый новый инструмент нужно писать отдельно;
- интеграции трудно переносить между клиентами;
- сложнее стандартизировать безопасность;
- сложнее объяснить, как вообще устроено подключение.

MCP решает это единым протоколом: один сервер можно подключить в разные клиенты.

## Как MCP подключается

Типовая схема выглядит так:

1. Вы описываете MCP-сервер.
2. Агент-клиент читает конфиг.
3. Клиент запускает локальный сервер или подключается к удаленному.
4. Сервер сообщает список инструментов.
5. Агент вызывает инструмент по протоколу.
6. Результат возвращается обратно в контекст.

## Local MCP

Local MCP — это сервер, который запускается на вашей машине как локальный процесс.

Когда local MCP удобен:

- когда инструмент должен работать с локальными файлами;
- когда не нужен внешний сетевой доступ;
- когда вы хотите максимальный контроль;
- когда задача требует простого и безопасного старта.

## Remote MCP

Remote MCP — это сервер, доступный по сети, часто через HTTP.

Когда remote MCP удобен:

- когда инструментом пользуются несколько клиентов;
- когда сервер уже размещен в инфраструктуре провайдера;
- когда вам нужен управляемый удаленный сервис;
- когда не хочется ставить локальную зависимость на каждую машину.

## Риски безопасности

MCP расширяет возможности агента, но добавляет риски.

- Каждый подключенный сервер расширяет доверенную поверхность.
- Инструменты могут получать доступ к данным и действиям, которые вы не планировали открывать.
- Локальные серверы могут читать локальные файлы.
- Удаленные серверы могут отправлять запросы наружу.
- Слишком много MCP-серверов увеличивает объем контекста и сложность контроля.

Что нужно делать на практике:

- подключать только те серверы, которые реально нужны;
- читать документацию сервера до подключения;
- ограничивать инструменты, если это возможно;
- использовать отдельные окружения для изоляции;
- проверять, какие данные уходят в удаленный сервис.

## Практические команды

Проверка конфигурации MCP в OpenCode:

```bash
cat ~/.config/opencode/opencode.json
```

Ожидаемый результат: в конфиге видно секцию `mcp` или становится понятно, что MCP еще не настроен.

Проверка справки OpenCode:

```bash
opencode --help
```

Ожидаемый результат: OpenCode выводит справку, значит клиент установлен и доступен из терминала.
